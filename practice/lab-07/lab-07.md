# Лабораторна робота №7. Поведінкові фактори та UX, Вебаналітика та SEO-стратегія

---

## Мета

Навчитись оцінювати SEO-ефективність сайту через поведінкові та аналітичні дані: проводити UX-аудит сторінок входу,
аналізувати CTR/Bounce/Engagement/Dwell-патерни, коректно налаштовувати GA4 події та конверсії, виконувати фінальний
SEO-аудит проєкту з пріоритезацією задач у roadmap.

---

## Якщо сайт недоступний онлайн (закінчився free trial)

Якщо публічний URL тимчасово недоступний, лабораторну можна виконати у двох форматах:

### Варіант A (рекомендований) - тимчасовий новий деплой

Розгорнути проєкт на безкоштовному хостингу (Vercel, Netlify, Render або інший free-tier) і виконати всі етапи
лабораторної на новому URL.

### Варіант B - аналітичний fallback

Якщо деплой неможливий, дозволено виконати лабораторну на історичних даних та демо-даних:

- UX-аудит і фінальний SEO-аудит - на основі попередніх лабораторних, DevTools/Lighthouse і існуючих артефактів
- Аналіз поведінкових показників - через експорт попередніх GSC/GA4 даних або GA4 Demo Property
- Налаштування GA4 - через GTM Preview + DebugView (демонстрація коректного спрацювання подій)
- Roadmap і пріоритезація - на основі зафіксованих evidence, навіть якщо частина KPI ще не має live-динаміки

### Що обов'язково вказати у звіті

1. Який формат виконання обрано: Варіант A або Варіант B
2. Якщо Варіант B - причина відсутності live URL
3. Які дані використано (період, джерело, обмеження)
4. Які перевірки виконані через DebugView/демо і як це підтверджено

### Чекліст виконання для Варіанту B (локально + демо-дані)

Заповнити окремий аркуш **"Fallback Mapping"** і зафіксувати, як саме закрито кожен пункт лабораторної:

| Розділ ЛР | Пункт | Що саме виконуємо | Формат (Локально / Демо-дані / Змішано) | Доказ |
|-----------|-------|-------------------|------------------------------------------|-------|
| 3. Налаштування GA4 | 3.2 Події для SEO-оцінки | Налаштовано `click_cta_primary` і `scroll_75` через GTM, перевірено в DebugView | Локально | Скрин GTM Preview + скрин DebugView з параметрами |
| ... | | | | |

Правила заповнення:

1. Для кожного пункту ЛР вказати лише один формат: `Локально`, `Демо-дані` або `Змішано`
2. Якщо формат `Демо-дані` - додати джерело (GA4 Demo Property / історичний експорт) і період
3. Якщо формат `Змішано` - коротко розділити, що зроблено локально, а що взято з даних
4. У колонці "Доказ" обов'язково вказати конкретний артефакт (скрін, експорт, посилання на аркуш)

## Завдання

### 1. UX-аудит сайту

#### 1.1 - Визначення пріоритетних landing pages

Обрати мінімум **10 сторінок входу з органічного трафіку** (або всі, якщо проєкт ще невеликий).

Заповнити таблицю **"Landing Audit"**:

| URL | Тип сторінки | Intent | Organic sessions (28 днів) | CTR (GSC) | Engagement rate (GA4) | Bounce context | Пріоритет |
|-----|--------------|--------|-----------------------------|-----------|-----------------------|----------------|-----------|
| https://example.com/articles/react-hooks-guide | article | Інформаційний | 540 | 3.8% | 58% | Високий bounce допустимий, але слабкий scroll | High |
| ... | | | | | | | |

**Мінімальна вимога:**

- мінімум 4 інформаційні URL
- мінімум 3 комерційні/категорійні URL
- мінімум 3 транзакційні або цільові URL (картка/форма/checkout/lead)

#### 1.2 - UX-чекліст першого екрана (above the fold)

Для мінімум **6 пріоритетних URL** виконати перевірку first screen:

| URL | Match між Title/H1/intent | Чітка цінність за 3-5 с | Помітний CTA | Елементи довіри | Mobile UX | Висновок |
|-----|---------------------------|-------------------------|--------------|-----------------|-----------|----------|
| https://example.com/category/frontend | Частково | Ні (довгий вступ без конкретики) | Так | Ні | Проблема з tap targets | Потрібен redesign hero-блоку |
| ... | | | | | | |

Критерії оцінки Mobile UX:

- tap targets не менші за `44x44 px`
- контент не ламається по ширині екрана
- критичні кнопки не перекриті sticky-елементами
- форма не перевантажена полями на першому кроці

## Контрольні питання

### Рівень 1 - Розуміння термінів

1. Чим відрізняються дані `до кліку` (GSC) і `після кліку` (GA4) у SEO-аналітиці?
2. Чому високий Bounce Rate не завжди означає поганий UX?
3. Що таке Dwell Time і як він пов'язаний із pogo-sticking?
4. Які події в GA4 є критичними для SEO-оцінки landing pages?
5. Навіщо сегментувати дані на brand/non-brand і mobile/desktop?

### Рівень 2 - Аналіз

6. У сторінки CTR 2.4% при позиції 4.8 і 20 000 показів. Які 3 гіпотези ви перевірите першими?
7. Non-brand кліки ростуть, але lead CR із органіки падає. Які причини найімовірніші за даними GA4?
8. На mobile engagement rate суттєво нижчий, ніж на desktop. Як визначити, чи проблема в UX першого екрана?
9. Які ризики виникають, якщо conversion events у GA4 налаштовані без стандартизованих параметрів?
10. Як інтерпретувати ситуацію: високий scroll depth, але низький клік по CTA?

### Рівень 3 - Синтез та висновки

11. Складіть власний топ-5 SEO-задач у форматі Impact/Effort для вашого проєкту і обґрунтуйте порядок.
12. Запропонуйте систему щотижневого SEO-review: які KPI, який ритм і хто owner для кожної метрики.
13. Опишіть, як ви доведете бізнесу ефект від UX-оптимізації, якщо позиції майже не змінились.
14. Сформуйте SMART-ціль на 90 днів для non-brand органічного трафіку та поясніть, які дані підтвердять успіх.
